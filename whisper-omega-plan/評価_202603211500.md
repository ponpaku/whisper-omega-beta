## 総評
現状は「設計思想をコードに落とした実働ベータ」として成立している。
さらに、ASR 本体については `faster-whisper` を使った end-to-end smoke test まで通っており、単なる骨格段階は超えている。

一方で、「WhisperX / faster-whisper 直利用から乗り換えさせる完成品」という目的にはまだ未達。
不足しているのは主に以下の3系統。

1. 本格機能の深さ
2. 導入・診断・GPU運用まわり
3. 仕様と実装の一致保証

---

## 1. すでにできていること
- CLI 骨格がある
  - `omega transcribe`
  - `omega whisperx`
  - `omega doctor`
  - `omega setup core`
- `success / degraded / failure` の実行モデルが入っている
- `runtime_policy`、`device`、`emit_result_json`、`write_failure_json` がコード化されている
- exit code が整理されている
  - 0 / 10 / 20 / 30 / 31 / 40
- machine-readable な failure / degraded JSON を返せる
- dependency missing 時に「落ちる」のではなく dependency failure を返せる
- `faster-whisper` による CPU smoke test が通っている
- `pyannote.audio` を使う diarization backend の実行経路が入っている
- `HF_TOKEN` 未設定時に diarization を configuration/degraded として返せる
- テストが通っている
- 仕様先行の構造になっており、将来的な拘束がしやすい

## 2. 未達の主要論点

### A. 機能面の未達
#### A-1. forced alignment が本実装になっていない
現状は best-effort で、WhisperX の主要価値である「本格的な forced alignment」の代替にはまだ足りない。

補足:
- ASR がすでに word timestamps を返すケースは alignment 成功として扱えている
- ただし、wav2vec2 系 backend による真の再アラインメントはまだ未実装

必要対応:
- 実際に wav2vec2 系や同等 backend を使った forced alignment を実装
- language ごとの align model 解決戦略を定義
- words がない場合の fallback ではなく、本当に alignment する
- alignment 成功/失敗時の metadata を契約どおりに残す

#### A-2. diarization backend は導入されたが、実利用完走には未達
pyannote backend はすでに現代的な `pyannote.audio` 前提に更新されている。
ただし、実利用には `HF_TOKEN` と追加の音声 decode 周辺整備が必要で、置き換え可能な完成度までは達していない。

必要対応:
- `HF_TOKEN` を含む実運用セットアップ手順を確定
- FFmpeg / torchcodec 周辺の依存を整理
- backend ごとの capability 宣言を導入
- diarization 未実行 / unavailable / degraded の境界を厳密化
- `none` / `pyannote` / 将来 backend の戻り値契約をそろえる

#### A-3. WhisperX 互換フロントが浅い
CLI 互換の入口はあるが、実質的なオプション吸収と挙動互換はまだ限定的。

必要対応:
- WhisperX 互換対象オプションを Appendix B/C に沿って絞り込む
- その対象については挙動・JSON・exit family を合わせる
- 受け流しだけの引数と、本当に意味を持つ引数を分離する
- 非対応引数は silent ignore せず warning / usage error / documented partial compatibility にする

---

### B. 導入・運用面の未達
#### B-1. `doctor` がまだ弱い
ここはプロダクト価値の中核なのに、現状は軽い環境確認に留まっている。

現状確認できるもの:
- Python / platform
- `faster-whisper` 利用可否
- `nvidia-smi` 利用可否
- GPU 名と driver

必要対応:
- CUDA driver
- cuDNN / cuBLAS の有無
- faster-whisper / ctranslate2 実行可能性
- HF token の有無
- diarization backend 利用可否
- alignment backend 利用可否
- cache dir / 書込権限
- known failure code への分類
まで診断対象を拡張する

#### B-2. `device=auto` が実質 auto になっていない
現状の `auto -> cpu` 固定は、製品としては未達。

必要対応:
- 利用可能 GPU がある場合は GPU を選ぶ
- `auto` と `cuda` の意味を厳密に分離する
- `--device cuda` は「GPU 必須」
- `--device auto` は「使えれば GPU、だめなら policy に従う」
- strict / permissive / strict-gpu と device の優先順位を実装で固定する

#### B-3. failure / degraded の observability は概ね入ったが、まだ強化余地がある
思想は正しく、主要な構造もすでに入っている。
ただし、運用で使うにはまだ強化余地がある。

必要対応:
- `error_code`
- `error_category`
- `backend_errors[]`
- `fallbacks[]`
- `requested_features`
- `completed_features`
- `failed_features`
をすべて契約どおり常時出せるようにする
- degraded 時の cause code ルールを実装で固定する
- CLI / JSON / Python API で意味論を一致させる

---

### C. 仕様と実装の一致保証が未完
#### C-1. JSON Schema をコード実装に結びつけ切れていない
仕様書はあるが、実装がそれを自動検証している状態までは行っていない。

必要対応:
- Appendix A の JSON Schema を正式化
- 生成結果を schema validate するテストを追加
- success / degraded / failure の代表 JSON fixture を作る
- nullability / enum / required field のずれを CI で検出する

#### C-2. 互換マトリクスと実装がまだ連動していない
Appendix B の存在だけでは足りない。

必要対応:
- 互換対象オプションを表で固定
- 各オプションに test case ID を付ける
- 同等 / 部分互換 / 非互換 をテストに反映
- WhisperX との差分理由と expected behavior を fixture 化する

#### C-3. Validation が本番受け入れレベルまで固定されていない
ベースライン方針はあるが、再現試験の粒度が不足。

必要対応:
- 固定モデル名
- 固定バージョン
- 固定サンプル音源
- 長さ別データセット
- 言語別データセット
- 故障注入ケース一覧
- benchmark 実行条件
を Appendix D に落とし、CI / 手動試験に接続する

## 4. 完成条件の具体化

### 完成とみなせる最低条件
- ASR が CPU/GPU 両方で実動する
- `device=auto` / `cpu` / `cuda` が意味どおりに動く
- strict / permissive / strict-gpu が仕様どおりに動く
- failure / degraded / success の JSON が schema validate を通る
- forced alignment が本当に動く
- diarization backend が少なくとも 1つ実用レベルで動く
- `doctor` が known failure をかなりの率で分類できる
- WhisperX 互換対象オプションのテストが通る
- Validation Protocol に沿った性能・診断・互換試験が再実行可能

### まだ「完成」と呼べない条件
- forced alignment が本実装でない
- `device=auto` が auto でない
- `doctor` が軽い
- schema と実装の一致が CI で担保されていない
- 互換対象がドキュメントだけで実装に落ちていない
- diarization が token / decode 環境整備なしで安定完走しない

## 6. 最終評価
現状は「思想が正しく、構造もよく、ベータとしては合格」。
さらに、ASR の実動と diarization backend の実行経路までは確認できている。

ただし「WhisperX の上位互換として移行を促せる完成品」にするには、
以下の4点を埋める必要がある。

- real functionality depth
- strong doctor / GPU diagnostics
- exact spec compliance
- compatibility depth

言い換えると、
今は「設計の勝ち」に加えて「一部実動の勝ち」までは取れているが、
まだ「製品の勝ち」には至っていない。

そのため、次の開発目標は
**ベータ基盤の整備完了**ではなく、
**実利用で置き換え可能な完成度への収束**
に置くべき。
